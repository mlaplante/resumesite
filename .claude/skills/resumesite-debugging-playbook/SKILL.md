# resumesite debugging playbook

A runbook, not a tutorial. Find your symptom, run the ONE discriminating command in
that row, then apply the fix. All facts below were verified directly against the
repo on 2026-07-05 (branch `skills`) â€” see "Provenance and maintenance" for how to
re-check anything that can drift.

**When NOT to use this skill:**
- The root cause is already known and documented â†’ read `resumesite-failure-archaeology`
  instead of re-deriving it.
- You're trying to measure performance, Lighthouse scores, or bundle size on a
  *working* site â†’ use `resumesite-diagnostics-and-tooling`.
- You need the general debugging method (bisection, hypothesis discipline, etc.)
  â†’ run `superpowers:systematic-debugging`, then return here for the project-specific
  table below.
- You're about to change behavior to fix something â†’ any fix still goes through a
  PR with green CI (see `resumesite-change-control`). Nothing here authorizes pushing
  straight to `master` or skipping CI.

## Before anything else: two known traps that cause confusing follow-on failures

| Trap | What it looks like | One command to confirm | Fix |
|---|---|---|---|
| **Two `node_modules` trees** | `tests/lib/*.test.js` fail to import, or `npm run typecheck`/`build` fail with missing modules, right after a fresh clone | `ls blog-src/node_modules` (errors if missing) | `npm ci` at repo root **and** `cd blog-src && npm ci`. The `lib` vitest project imports `blog-src/src/utils/*.ts` (needs `pdfkit`, `wawoff2` from blog-src's install) and the tsconfig `extends` chain (`astro/tsconfigs/strict`). Root-only `npm ci` is not enough. |
| **`blog-src/` naming trap** | You go hunting for the homepage/rĂ©sumĂ©/services pages somewhere else because "blog-src" sounds blog-only | `ls blog-src/src/pages` | The entire Astro site â€” home, rĂ©sumĂ©, services, contact, privacy, `/uses`, 404 â€” lives under `blog-src/`, not just the blog. Only `blog-src/scripts/purge-css.mjs`, `blog-src/package.json`, and the site config are truly "just build tooling"; everything else is the whole site. |

## Symptom â†’ cause â†’ check â†’ fix

| Symptom | Likely cause | Discriminating check | Fix |
|---|---|---|---|
| Page script silently doesn't run (no console error visible to the user, or a CSP violation in DevTools console: `Refused to execute inline script...`) | New `<script>` block violates the CSP `script-src` allowlist, OR Astro inlined a small `<script type="module">` bundle as `is:inline` HTML because `assetsInlineLimit` isn't 0 | `grep -n "script-src" blog-src/public/_headers` then `grep -n assetsInlineLimit blog-src/astro.config.mjs` | `script-src` in `blog-src/public/_headers` has **no `'unsafe-inline'`** â€” that's deliberate (see `resumesite-change-control`, never relax it). Existing page scripts are all classic scripts loaded via `<script src="/js/...">` (see `blog-src/src/layouts/*.astro`, `BaseHead.astro`) â€” follow that pattern, don't add inline `<script>` bodies. A new *external* script origin (analytics, a widget) must be added to `script-src` deliberately, reviewed in the PR. `vite.build.assetsInlineLimit: 0` in `blog-src/astro.config.mjs` exists specifically so Astro never inlines a bundled script under the CSP; if someone reverts that setting, small module scripts silently start getting embedded inline and get blocked. |
| `npm run build` (or `npm run typecheck`) fails on a specific post/draft file | Zod schema violation in `blog-src/src/content.config.ts`: `title` empty or >200 chars, `excerpt` empty or >300 chars, unparseable `date`/`updated`, empty `category`, or `series` set to an empty string | Run `npm test` first â€” `tests/site/content.test.ts` mirrors the same schema bounds and runs in the fast `site` vitest project without a full build, so it usually catches the exact file+field. If it doesn't, `npm run typecheck` (`cd blog-src && npx astro check`) prints the exact Zod error. | Fix the frontmatter field named in the error. Reminder: `excerpt` is 1â€“300 chars (it becomes the OG/meta description); `title` is 1â€“200 chars. Drafts under `blog-src/src/content/drafts/` are NOT validated by the build (the loader glob only covers `content/posts/`) â€” a bad draft won't fail CI until it's moved to `posts/`. |
| `tests/site/build-output.test.ts` shows `1 skipped` (or the whole `site` describe block is skipped) | This is **expected**, not a bug, for a plain local `npm test` â€” the file self-skips via `describe.skipIf(!hasDist)` when `dist/` doesn't exist | `ls dist` (errors if absent â€” self-skip is correct; succeeds if a build already ran and the test should NOT be skipping) | If you need it to actually assert, run a build first: `cd blog-src && npm run build`, then `npx vitest run --project site` from repo root. This populates enough of `dist/` for these specific assertions, but it is **not** the full pipeline CI runs (root `npm run build` also runs PurgeCSS and asset fingerprinting) â€” for the complete build pipeline and PurgeCSS/fingerprinting issues, see `resumesite-build-and-env`. Do **not** run the root `npm run build` from the repo root during a debugging session unless you intend to â€” it deletes any `blog-src/src/**/CLAUDE.md` and writes `dist/`. |
| `tests/site/build-output.test.ts` fails (not skipped) after a real build | The build didn't render one of: OG images (`dist/og/*.png`), the rĂ©sumĂ© PDF (`dist/resume.pdf`, must start with `%PDF-` and be >20KB), or the search index (`dist/blog/search.json`, must be a non-empty array) â€” or a required route/file is missing from `dist/` | The test names tell you exactly which required path or asset failed (`it.each(required)`, `'generates a per-post Open Graph card...'`, `'produces a real (non-empty) PDF rĂ©sumĂ©'`, `'builds a populated client-side search index'`) | Re-run `cd blog-src && npm run build` and read the astro/satori/pdfkit output for a rendering error; these are all build-time renderers (satori + @resvg/resvg-js for OG, pdfkit for the rĂ©sumĂ©). A missing route file usually means a page in `blog-src/src/pages` was renamed/removed without updating the `required` list in the test or the route itself. |
| Contact form POST returns **403** | Same-origin enforcement: `Origin` header â‰  `url.origin` â€” **including a POST with no `Origin` header at all** (this is deliberate â€” it closed a prior cross-site bypass; full history in `resumesite-failure-archaeology`) | `grep -n "origin !== url.origin" worker/api/contact.ts` | Not a bug by default â€” it's the fixed cross-site bypass. Only investigate if a *legitimate* same-origin browser POST is hitting this: check that the request truly came from a `<form>` served by this origin (not proxied/rewritten by a CDN or test harness that drops `Origin`). |
| Contact form POST returns **415** | `Content-Type` header doesn't start with `application/x-www-form-urlencoded` or `multipart/form-data` | `grep -n "Unsupported media type" worker/api/contact.ts` | Fix the client to submit as an actual HTML form POST (browsers do this by default); don't POST JSON to `/api/contact`. |
| Contact form POST returns **413** | Body over `MAX_FORM_BYTES` (32 KiB) â€” checked twice: once cheaply against the (client-controlled) `Content-Length` header, then for real against the streamed body via `readBodyCapped` (catches chunked uploads with no `Content-Length`) | `grep -n MAX_FORM_BYTES worker/api/contact.ts` | Message field is capped at `MAX_MESSAGE_LEN = 5000` chars downstream anyway â€” if a legitimate long message trips this, that's a product decision (raise `MAX_FORM_BYTES`/`MAX_MESSAGE_LEN` via change-control), not a bug. |
| Contact form POST returns **400** | `new Response(rawBody, {headers}).formData()` threw â€” malformed multipart boundary or corrupt urlencoded body | `grep -n "Bad request" worker/api/contact.ts` | Almost always a malformed request from a script/tool hitting the endpoint directly, not a real form submit. Verify with a real browser form POST. |
| Contact form "succeeds" (303 â†’ `/thank-you/`) but nothing shows up in D1/email | `bot-field` was non-empty â†’ honeypot fired; the worker **silently pretends success** (redirects to `/thank-you/`) and never touches D1 or Turnstile/ForwardEmail | `grep -n "bot-field" worker/api/contact.ts` | This is intentional anti-spam behavior, not a bug â€” don't "fix" it to show an error. If a *real* user is being caught by the honeypot, check the form markup for `name="bot-field"` and make sure it's still properly hidden/empty for humans (CSS/off-screen), not renamed or exposed. |
| Contact form validation/turnstile/rate-limit failures all look like "it just redirects to an error page" with no HTTP error code | By design: only the **pre-parse structural** guards (origin/content-type/size/malformed-body) return real HTTP status codes (403/415/413/400). Everything after body parsing â€” empty name/message, bad email regex, missing/invalid Turnstile token, Turnstile network failure, rate limit â€” goes through the `reject()` helper, which **always returns 303 â†’ `/contact-error/`**, logging `console.warn('contact: rejected on <reason>', details)` | `grep -n "rejected on" worker/api/contact.ts` then check Worker logs (`wrangler tail` in prod, or the test's console output) for the exact `reason` string | Not a bug â€” read the `reason` in the warning log to know which guard fired (`validation`, `missing turnstile token`, `turnstile`, `turnstile verify error`, `rate limit`). There is no 429 anywhere in this handler. |
| Contact form always rate-limited when running the **full** test suite, but the same test passes when run **alone** | `vitest.config.ts`'s `worker` project uses `cloudflareTest({ singleWorker: true, ... })` â€” one shared miniflare D1 instance for the whole `worker` project, not per file. `tests/worker/contact.test.ts` clears `submissions`/`contact_attempts` in its own `beforeEach`, but that `beforeEach` only runs for tests **in that file**. A new test file that also POSTs to `/api/contact` (reusing the default test IP `203.0.113.5`) without its own table-clearing `beforeEach` inherits whatever rows contact.test.ts's last DB-touching test left behind. | Run the new/failing test file alone: `npx vitest run --project worker path/to/new.test.ts` vs. the whole project: `npx vitest run --project worker`. If it passes alone and fails as part of the full run, it's shared D1 state, not a real rate-limit bug. | Add the same `beforeEach(() => { DELETE FROM submissions; DELETE FROM contact_attempts; })` pattern from `tests/worker/contact.test.ts` to the new file, or use a distinct fake `CF-Connecting-IP` per test file. |
| `tests/lib/*.test.js` fail to import (`Cannot find module`, `Cannot find package 'pdfkit'`, etc.) | `blog-src/node_modules` missing â€” the `lib` vitest project (plain Node, `environment: 'node'`) imports `blog-src/src/utils/*.ts` (e.g. `tests/lib/resume-pdf.test.js` imports `blog-src/src/utils/resumePdf.ts` and `blog-src/src/data/resume.ts` directly) | `ls blog-src/node_modules` (errors if missing) | `cd blog-src && npm ci`. CI does this too â€” see `.github/workflows/ci.yml`'s `test` job: root `npm ci`, then `npm ci` again with `working-directory: blog-src`, *then* `npm test`. |
| `scripts/generate-post*.js` (or the `blog:draft:*` npm scripts / the daily `generate-blog-post.yml` cron) aborts with `Could not find a sufficiently unique topic after 4 attempts.` | The topic picker (`pickUniqueTopic` in `scripts/lib/blog-post.js`) rejected 4 straight candidates as too similar to an existing post | Check which script/provider is running: `scripts/generate-post.js` (root `npm run blog:draft:*`, Anthropic, **no `embed` adapter passed â†’ lexical Jaccard only**, `DUPLICATE_THRESHOLD = 0.5`) vs. `scripts/generate-post-gemini.js` (the daily CI cron, **has an `embed` adapter â†’ semantic cosine**, `SEMANTIC_THRESHOLD` env-tunable, default `0.85`). Console output right before the abort shows `Rejected candidate "..." â€” NN% <lexical\|semantic> similarity to existing "..."` for each attempt. | This is a dedupe-threshold tuning question, not a code bug â€” see `resumesite-blog-quality-campaign` for the dedupe design and how/whether to adjust `SEMANTIC_THRESHOLD`/`DUPLICATE_THRESHOLD`. Don't hand-tune thresholds outside that skill's guidance. |
| CI's Lighthouse step (`.github/workflows/ci.yml`, `site` job, "Lighthouse audit") fails the job | `categories:accessibility` scored below `0.90` on one of the 5 audited pages (`/index.html`, `/blog/index.html`, `/blog/hello-world/index.html`, `/services/index.html`, `/resume/index.html` â€” see `lighthouserc.json`). Performance/best-practices/SEO regressions only `warn`, they never fail the job. | `grep -n "categories:accessibility" lighthouserc.json` â€” confirms `["error", {"minScore": 0.9}]` is the only hard-fail assertion | Fix the actual a11y regression (contrast, missing labels, ARIA, etc.) on the specific page named in the LHCI report. For how to read/reproduce the Lighthouse report locally, see `resumesite-diagnostics-and-tooling`. |
| `curl`ing the live site (or the deploy-verify job) gets an unexpected **403** on a path that should work | You probed the **custom domain** (`michaellaplante.com`) from a CI runner / script â€” Cloudflare's zone-level bot protection challenges automated requests before they reach the Worker | Compare: does the same path succeed against the `workers.dev` host? See `.github/workflows/purge-cloudflare-cache.yml`'s `verify` job â€” it deliberately probes `${DEPLOY_CHECK_HOST:-resumesite.laplantewebdevelopment.workers.dev}`, not the custom domain, for exactly this reason. `GET /api/contact` there should return **405** (proves the Worker is deployed and routing; a missing Worker would 404/5xx instead). | Not a bug in the site â€” expected WAF behavior for the custom domain against automated clients. If the account's `workers.dev` subdomain ever changes, update the `DEPLOY_CHECK_HOST` repository *variable* (not a secret), not the workflow file. |
| Draft-generation embedding step is slow / re-embeds far more than expected on one run | `scripts/.embeddings-cache.json` is **gitignored** (see `.gitignore`) â€” a fresh clone, a fresh CI runner, or a locally-deleted cache file means the next run embeds every existing post's title+excerpt fresh (111 posts as of 2026-07-05, `ls blog-src/src/content/posts/*.md \| wc -l`; the code comment in `scripts/lib/blog-post.js` describes this generically as "70+") instead of just the new candidate, because the cache is what limits a normal daily run to ~1 embedding call | `ls scripts/.embeddings-cache.json` (errors if missing â€” next run re-embeds everything) | Expected, self-healing cold start â€” not a bug. Only `generate-post-gemini.js` and `generate-post-gh-models.js` populate/use this cache (they pass an `embed` adapter into `runGenerator`); `generate-post.js` (Anthropic) never touches it and always uses the lexical fallback. The daily cron (`generate-blog-post.yml`) runs `generate-post-gemini.js`, so it's the one affected by cache loss. |
